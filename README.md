# MarketPulse

Бэкенд многопользовательского (multi-tenant) SaaS-сервиса, который помогает продавцам
на Wildberries, Uzum Market и других маркетплейсах отслеживать производительность
складского персонала в реальном времени — от сканирования штрихкода до автоматического
расчёта зарплаты.

Это переписанный с нуля прототип, изначально построенный с production-практиками:
корректное разделение окружений, тесты, Docker и расширяемая архитектура для добавления
новых маркетплейсов.

## Что умеет проект

- Подключается к аккаунту продавца на Wildberries и **автоматически синхронизирует каталог
  товаров в фоне**, каждую минуту, через Celery.
- Позволяет складским работникам **сканировать штрихкод** сразу после упаковки товара
  через простой API-эндпоинт.
- Сопоставляет каждое сканирование с синхронизированным каталогом и **помечает повторные
  сканирования за тот же день** вместо того, чтобы молча их отбрасывать.
- Даёт владельцам/менеджерам **ежедневную сводку по продуктивности и зарплате**: кто, сколько
  единиц упаковал и сколько заработал — рассчитывается автоматически по настраиваемой ставке
  за единицу.
- **Изначально многопользовательский (multi-tenant)**: каждый бизнес — это `Organization` со
  своими складами, пользователями и подключениями к маркетплейсам — готов к работе с
  несколькими клиентами одновременно.
- **Рассчитан на несколько маркетплейсов**: чтобы добавить Uzum Market или другую платформу,
  достаточно написать один новый класс-клиент, не трогая остальной код.

## Архитектура

```
                    ┌─────────────┐
                    │   Django    │   REST API (DRF) + админка
                    │ (веб-прил.) │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        ┌─────▼────┐ ┌─────▼─────┐ ┌────▼─────┐
        │PostgreSQL│ │   Redis   │ │  Celery   │
        │          │ │ (брокер)  │ │ worker +  │
        │          │ │           │ │  beat     │
        └──────────┘ └───────────┘ └─────┬─────┘
                                          │  каждую минуту
                                    ┌─────▼──────┐
                                    │  API       │
                                    │ маркетпл.  │
                                    │ (WB, Uzum..)│
                                    └─────────────┘
```

**Приложения:**

| Приложение | Ответственность |
|---|---|
| `accounts` | Кастомная модель пользователя, роли (владелец / менеджер / работник), аутентификация (JWT) |
| `organizations` | Модель multi-tenant: `Organization` и её `Shop` (склады) |
| `integrations` | Подключения к маркетплейсам, синхронизированный каталог `Product`, Celery-задачи синхронизации, подключаемые клиенты маркетплейсов |
| `scanning` | События сканирования штрихкодов, обнаружение дубликатов, сводка по продуктивности и зарплате |

## Технологический стек

- **Backend:** Python 3.14, Django 6.0, Django REST Framework
- **База данных:** PostgreSQL (продакшен/Docker), SQLite (локальная разработка без Docker)
- **Фоновые задачи:** Celery + Redis, `django-celery-beat` для управления расписанием
- **Аутентификация:** JWT (`djangorestframework-simplejwt`)
- **Тесты:** pytest, pytest-django, factory-boy, coverage
- **Качество кода:** black, isort, flake8, GitHub Actions CI
- **Контейнеризация:** Docker + docker-compose

## Быстрый старт (Docker — рекомендуется)

```bash
git clone https://github.com/your_username/marketpulse.git
cd marketpulse
cp .env.example .env        # укажите реальные значения, особенно SECRET_KEY и пароль БД
docker-compose up --build
```

Это поднимет: Django-приложение (`localhost:8000`), PostgreSQL, Redis, Celery worker и
Celery beat (который запускает синхронизацию с маркетплейсом каждую минуту).

Создайте администратора:

```bash
docker-compose exec web python manage.py createsuperuser
```

Откройте `http://localhost:8000/admin/` для входа или `http://localhost:8000/health/`,
чтобы убедиться, что API работает.

## Быстрый старт (без Docker)

```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements-dev.txt
cp .env.example .env
python manage.py migrate
python manage.py runserver
```

Используется SQLite, без Redis/Celery — этого достаточно, чтобы изучить API, но фоновая
синхронизация с маркетплейсом без Redis и запущенного воркера Celery работать не будет.

## Запуск тестов

```bash
pytest --cov=. --cov-report=term-missing
```

Обращения к API маркетплейсов в тестах замоканы — реальный ключ Wildberries API для
запуска тестов не требуется.

## Обзор API

| Эндпоинт | Метод | Описание |
|---|---|---|
| `/api/token/` | POST | Получить пару JWT-токенов (access/refresh) |
| `/api/accounts/register/` | POST | Создать нового пользователя |
| `/api/accounts/me/` | GET | Профиль текущего пользователя |
| `/api/organizations/shops/` | GET/POST | Список/создание складов (в рамках своей организации) |
| `/api/scanning/events/` | GET/POST | Список сканирований / запись нового сканирования штрихкода |
| `/api/scanning/events/summary/` | GET | Сводка по продуктивности и зарплате за сегодня по каждому работнику |

## Заметки по безопасности

- Секреты в репозиторий не коммитятся. `.env` в `.gitignore`; в репозитории хранится только
  `.env.example` с заглушками.
- API-ключи маркетплейсов хранятся только на сервере (`MarketplaceConnection.api_key`) и
  никогда не отдаются через API или в списочных представлениях админки.
- `SECRET_KEY` Django обязательно нужно сгенерировать заново для любого реального
  развёртывания — никогда не используйте заглушку из `.env.example`.

## Известные ограничения / компромиссы

Пишем об этом открыто, поскольку README без единого «известного недостатка» обычно
выглядит подозрительно:

- **Ставка оплаты не сохраняется по историю.** `ScanEvent.summary` рассчитывает заработок
  по *текущей* `Product.rate_per_unit` на момент запроса, а не по той ставке, что
  действовала на момент сканирования. Если ставка меняется, прошлые сводки будут
  отражать новую ставку. Правильное решение — сохранять ставку прямо в `ScanEvent` в
  момент сканирования; оставлено как ближайшее улучшение, а не сделано преждевременно.
- **`MarketplaceConnection.api_key` хранится в базе данных в открытом виде.** Это приемлемо
  для демонстрационного/портфолио-проекта; в реальном продакшене это поле нужно
  шифровать (например, `django-fernet-fields` или менеджер секретов) перед работой с
  реальными учётными данными продавцов.
- **Разбор ответа клиента Wildberries основан на форме публичного API на момент
  написания** и должен быть перепроверен по актуальной документации Wildberries API
  перед использованием в продакшене — маркетплейсы меняют контракты API без особого
  предупреждения.

## Участие в разработке

Проект структурирован так, чтобы небольшая команда могла работать параллельно: каждое
Django-приложение самодостаточно — со своими моделями, тестами и маршрутами API. Перед
открытием PR:

```bash
black . && isort . && flake8 .
pytest
```

CI выполняет те же проверки при каждом push.
